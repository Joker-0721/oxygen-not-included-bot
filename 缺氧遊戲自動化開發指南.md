# 《缺氧》自主代理人控制系統開發協議 (Development Protocol)

## 1. 核心研究目標 (Major Objective)
本項目旨在開發一個基於「觀察-推理-行動」(ReAct) 框架的自動化系統 [1, 2]。主要目標是建立一個跨語言的底層架構，使外部 AI 模型（如 Gemini 或 GPT-4）能夠透過結構化數據實時監控《缺氧》的遊戲狀態，並利用 C# 遊戲模組注入技術，精確執行微觀建築指令與宏觀資源調度 [3, 4]。

---

## 2. 階段性小目標與實施步驟 (Sub-Goals & Execution)

### 目標一：建立開發與調試基礎環境
**說明**：在不修改原始碼的前提下，利用 Harmony 補丁技術建立實驗平台。
*   **關鍵操作**：配置.NET 4.7.2 環境，使用 dnSpy 分析 `Assembly-CSharp.dll` 尋找關鍵類 [4, 5]。
*   **AI 溝通提示詞 (Prompt)**：
    > 「請扮演資深 Unity 遊戲 Modder。我正在為《缺氧》開發自動化項目，請提供一個 C# 模組模板，要求使用 Harmony 補丁庫。模組需要在遊戲啟動時載入，並在控制台列印 'Automation Link Initialized'。請說明如何引用遊戲核心 DLL 文件並設置輸出路徑。」[6, 4]

### 目標二：世界數據解耦與結構化導出 (感知層)
**說明**：將遊戲中複雜的磁磚（Tile）數據與實體數據轉化為 AI 可理解的矩陣或 JSON。
*   **關鍵操作**：讀取靜態類 `Grid` 中的 `Temperature`、`Mass` 和 `Element` 數組 [7, 8]。
*   **AI 溝通提示詞 (Prompt)**：
    > 「我需要從《缺氧》中提取當前基地的物理數據。請寫一段 C# 代碼，攔截遊戲的主循環（如 `GameUpdater.Update`），並每隔 5 秒獲取玩家當前視野中心 20x20 磁磚的數據。數據應包含每個 Cell 的元素 ID、溫度（Kelvin）以及質量。請將這些數據封裝成一個可序列化的 JSON 對象。」 

### 目標三：指令集抽象化與執行 API (行動層)
**說明**：將玩家的滑鼠點擊動作轉化為程序化 API。
*   **關鍵操作**：調用 `BuildTool.Instance.TryBuild` 進行建造，以及 `DigTool.Instance.AddItem` 進行挖掘 [9, 10]。
*   **AI 溝通提示詞 (Prompt)**：
    > 「請分析《缺氧》的 `BuildTool` 類。我希望寫一個 C# 函數 `InvokeBuild(string buildingId, int cellIndex, string materialId)`。請示範如何正確模擬玩家點擊，在特定座標發布建造指令，並確保該指令會被加入到小人的任務隊列（Errands）中。」 [4, 10, 11]

### 目標四：跨語言通訊橋樑 (傳輸層)
**說明**：在 C# (遊戲端) 與 Python (AI 代理端) 之間建立低延遲的 Socket 連接 [12, 13]。
*   **關鍵操作**：使用 UDP 或 TCP 協議，將感知數據傳給 Python 並接收行動指令 [14, 12]。
*   **AI 溝通提示詞 (Prompt)**：
    > 「建立一個 Python 腳本作為控制中心。它需要開啟一個 UDP 端口來接收來自 C# 模組的 JSON 數據流。請實現一個簡單的邏輯：如果接收到的 JSON 顯示基地平均氧氣分壓低於 500g，則透過 Socket 返回一個字符串指令 'BUILD_ALGAE_TERRARIUM' 到遊戲端。」 [14, 13]

### 目標五：建立多代理人邏輯決策框架 (智控層)
**說明**：運用 Multi-Agent 模式分工，分別處理電力、食物與生命維持 [14, 15]。
*   **關鍵操作**：定義 Prompt 中的物理規則，告知 AI 《缺氧》中的熱力學與元素位移特點 [16, 17]。
*   **AI 溝通提示詞 (Prompt)**：
    > 「你現在是『生命維持專家代理人』。你的決策環境是《缺氧》。注意遊戲規則：1. 一格只能存在一種元素。2. 二氧化碳會向下堆積。3. 熱量只通過傳導傳播。
    > 現在輸入感知數據：{JSON_DATA}。請評估當前氧氣供應的穩定性，如果需要行動，請以 JSON 格式給出行動座標和建築類型，並解釋你的鏈式思考路徑（Chain-of-Thought）。」 

### 目標六：自動化測試與穩健性優化
**說明**：利用調試模式加速測試，解決內存洩漏與效能瓶頸 [18, 19]。
*   **關鍵操作**：啟用 `debug_enable.txt`，並監控自動化運行後的 FPS 變化 [19, 20]。
*   **AI 溝通提示詞 (Prompt)**：
    > 「自動化代碼在遊戲運行 2 小時後導致內存佔用達到 15GB 且幀率下降。請檢查我的 C# 數據採樣器代碼，是否存在未釋放的對象引用或過於頻繁的反射調用？請提供一個緩存機制優化建議，只在數據發生顯著變化時才進行序列化。」 [21, 18, 20]

---

## 3. 技術風險控管
1.  **數據過載**：全圖 15 萬個磁磚數據會拖慢通訊，建議僅導出「活動區域」 [22, 18]。
2.  **版本更新**：Klei 的 QOL 更新頻繁，建議將 API 調用封裝在中間層，以便隨時更新 DLL 引用 [23, 24]。
3.  **邏輯死鎖**：若 AI 指令與小人優先級衝突，可能導致任務停滯。需定期監控小人壓力值與路徑不可達（Unreachable）通知 [25, 26]。

---

## 4. 總結
通過將《缺氧》的底層物理矩陣導出，並與 Python 層的大型語言模型對接，本項目的開發者可以利用 Prompt Engineering 賦予 AI 工程學知識。這種「解碼-決策-編碼」的循環是實現全自動殖民地管理的唯一路徑。

# 針對《缺氧》系統模擬環境之自動化代理人開發框架技術報告

## 自動化遊戲開發之系統架構與複雜度分析

在當前人工智慧與自動化技術蓬勃發展的背景下，針對《缺氧》（Oxygen Not Included, ONI）這類具有極高系統複雜度的模擬遊戲開發自動化玩遊戲代碼，其本質上是一項跨學科的工程挑戰。這不僅涉及傳統的電腦視覺與輸入模擬，更深層次地要求開發者建立起一套能夠與大型語言模型（LLM）代理人高效溝通的技術基礎設施。這類項目的核心在於如何將一個基於磁磚（Tile-based）的、遵循特定物理規則的二維模擬世界，轉化為 AI 代理人可以理解、推理並操作的結構化數據流。

《缺氧》的遊戲引擎由兩大核心層次組成：底層的物理模擬層（Physics Sim）主要由 C++ 編寫，負責氣體、液體的流動、熱傳導及壓力計算；而上層的邏輯層（Logic Layer）則由 C# 編寫，負責小人（Duplicants）、生物（Critters）、建築及任務調度。開發自動化程序的首要步驟，是建立一個能夠雙向溝通的橋樑，這通常需要結合 C# 模組開發（Modding）以獲取內存數據，以及 Python 腳本進行決策優化。

## 核心技術工具鏈之建構與整合

要實現高效的自動化開發，必須構建一套工業級的工具鏈。這套工具鏈應具備靜態代碼分析、動態內存攔截及視覺特徵識別等功能，以確保 AI 代理人能夠獲得實時且精確的環境反饋。

## C# 模組開發與 Harmony 補丁技術

《缺氧》是基於 Unity 引擎開發的，其核心邏輯封裝在 Assembly-CSharp.dll 檔案中。為了讀取遊戲狀態（如氧氣含量、溫度、資源量）並發布指令（如挖掘、建造），開發者必須掌握 C# 反編譯與補丁技術。

| 工具名稱 | 專業功能說明 | 應用場景 |
| :---- | :---- | :---- |
| **dnSpy / ILSpy** | C# 反編譯器與調試器 | 閱讀遊戲源代碼，尋找關鍵屬性（如 Game.Instance）的內存地址。 |
| **Harmony** | 運行時補丁庫（Patching Library） | 在不修改原始檔案的情況下，攔截並替換遊戲方法邏輯。 |
| **Visual Studio** | 集成開發環境（IDE） | 編寫.NET Framework 4.7.2 兼容的 C# 模組代碼。 |
| **Unity Mod Tools** | 場景層次結構查看器 | 查看遊戲運行時的物件樹狀結構，定位 UI 或建築實例。 |
| **ILMerge / Publicizer** | 類庫整合與私有欄位公開工具 | 簡化與遊戲私有成員的交互，解決編譯時的引用問題。 |

Harmony 提供的「Transpiler」補丁技術對於自動化尤為重要，它允許開發者直接修改 IL 指令（Intermediate Language），這在需要攔截極高頻率調用的物理循環（如熱傳導計算）時，比一般的「Prefix」或「Postfix」更具效能優勢。

## Python 視覺層與輸入模擬

雖然模組開發能提供精確數據，但有些動態 UI 或圖像特徵（如特定生物的動畫狀態）可能更適合透過視覺識別處理。此外，Python 擁有目前最強大的 AI 代理人框架（如 LangChain 或 Vertex AI），是溝通層的最佳選擇。

| 庫名稱 | 技術角色 | 主要用途 |
| :---- | :---- | :---- |
| **OpenCV** | 電腦視覺處理 | 進行模板匹配，識別屏幕上的警告圖標或特定建築物。 |
| **PyAutoGUI** | 跨平台 GUI 自動化 | 模擬滑鼠移動、點擊與鍵盤輸入，執行建造與菜單導航。 |
| **Pillow (PIL)** | 圖像數據處理 | 處理屏幕截圖，為 OpenCV 提供預處理的數據流。 |
| **Socket / asyncio** | 通訊協議層 | 在 C# 遊戲插件與 Python AI 代理人之間建立低延遲通訊。 |

## 遊戲數據獲取與內存映射機制

為了讓 AI 代理人能夠「思考」，必須將遊戲內的二維矩陣數據結構化。在《缺氧》中，世界是由磁磚組成的，每個磁磚包含元素 ID、質量、溫度及壓力等核心屬性。

## 全局實例與世界數據提取

開發者應重點關注 Game.Instance、World.Instance 以及 Grid 靜態類。Grid 類提供了一系列的數組，如 Grid.Temperature、Grid.Mass 和 Grid.Element，這些數組存儲了全圖數萬個磁磚的實時物理數據。

透過 C# 模組，開發者可以實現一個定時採樣器（Sampler），例如每 10 秒將當前基地核心區域的數據序列化為 JSON 格式。這種結構化數據是與 AI 溝通的基石。考慮到遊戲後期可能存在超過 15 萬個磁磚，傳輸全圖數據會導致嚴重的效能瓶頸（甚至使幀率降至 20fps 以下），因此應採取區域採樣（Local ROI）策略。

## 任務隊列與小人優先級監控

自動化的另一個關鍵是任務調度。遊戲內的任務（Errands）由一個複雜的隊列管理，受到小人的技能、興趣以及玩家設定的優先級（1-9）共同影響。模組應導出當前未完成任務的清單及其所需的技能標籤，這能讓 AI 代理人判斷是否需要調整優先級，或者是否因為路徑不通（Unreachable）導致進度停滯。

## 建立 AI 代理人通訊橋樑：數據協議與指令集

在開發過程中，如何使 AI 代理人理解「當前情況」並給出「操作建議」是項目的核心難點。這需要建立一套精細的溝通協議。

## 模型上下文協議（MCP）的應用

建議採用 Google 或 Anthropic 推廣的模型上下文協議（Model Context Protocol, MCP），這是一種標準化的通訊方式，允許 AI 模型透過 Get、Invoke、List 等函數直接調用外部工具（如獲取遊戲截圖、查詢磁磚溫度或發布建造指令）。

## 建立專用指令集（Action Space）

開發者應將複雜的遊戲操作抽象化為一組簡單的指令集，供 AI 代理人調用：

1. **建造指令**：Build(building_id, x, y, material_id, priority)。
2. **挖掘指令**：Dig(x1, y1, x2, y2, priority)。
3. **優先級調整**：SetPriority(task_type, value)。
4. **研究調度**：Research(tech_id)。

這種抽象化減少了 AI 的推理負擔，使其能專注於宏觀策略，如「熱量管理」或「氧氣循環優化」。

## 《缺氧》物理規則的 AI 建模指導

要與 AI 代理人溝通，開發者必須在提示詞（Prompts）中精確定義遊戲的物理定律。由於《缺氧》的模擬並非完全遵循現實物理，這些「偏離現實」的規則必須明確告知 AI。

## 熱力學模擬規則

熱量在《缺氧》中主要透過傳導傳播，不具備真實的熱對流（熱空氣不會自動上升）。

在遊戲中，傳熱速率受材料的導熱係數（Thermal Conductivity）與比熱容（Specific Heat Capacity）影響。AI 需要理解「熱量刪除」（Heat Deletion）的概念，例如透過電解器產生的氣體溫度雖然高，但其總熱能可能低於輸入的水，這在熱力學上是不守恆的，卻是遊戲生存的關鍵。

## 元素單一性與位移規則

磁磚世界遵循「一格一元素」原則（One element per cell）。這意味著兩種氣體無法在同一格混合。當多種元素競爭空間時，會發生位移（Displacement）。AI 必須理解這一點，才能設計出高效的「液體鎖」（Liquid Lock）或「無窮存儲」（Infinite Storage）。

## 開發步驟：從環境搭建到閉環自動化

完成工具選擇後，應遵循系統化的開發流程，確保 AI 代理人與遊戲環境的深度耦合。

## 第一階段：數據採集與觀測環境建立

開發者應首先建立一個測試環境。在《缺氧》中，這意味著啟用「調試模式」（Debug Mode）。

1. 在遊戲資料夾下創建 debug_enable.txt。
2. 利用 Ctrl+F4 開啟即時建築，Ctrl+U 加速時間，快速測試自動化邏輯。
3. 開發一個 C# 模組，攔截 GameUpdater.Update 方法，每秒導出一次基地的「生命體徵」數據。

## 第二階段：溝通層與推理循環實現

這一步是將 AI 模型（如 Gemini 1.5 Pro 或 GPT-4o）整合進來。建議採用 ReAct（Reasoning and Acting）設計模式。

* **思考（Thought）**：AI 分析數據（如「二氧化碳在基地底部堆積」）。
* **行動（Action）**：AI 發布指令（如「在座標 (20, 5) 建造碳素脫離器」）。
* **觀察（Observation）**：模組反饋指令執行結果（如「電力不足，建築未運作」）。

## 第三階段：專長代理人協作（Multi-Agent System）

對於《缺氧》這類需要同時處理電力、氧氣、食物、熱量及工業生產的遊戲，單一 AI 代理人可能會發生邏輯崩潰。應建立「專家團隊」架構：

| 代理人角色 | 核心職責 | 關鍵反饋數據 |
| :---- | :---- | :---- |
| **生命維持專家** | 確保氧氣與食物穩定 | 氧氣分壓、食物卡路里總量。 |
| **能源架構師** | 管理電網負載與燃料 | 電池電量、發電機負荷係數。 |
| **工業與熱能專家** | 負責金屬精煉與冷卻迴路 | 冷卻劑溫度、發熱建築狀態。 |
| **殖民地調度員** | 優化小人任務路徑與優先級 | 任務成功率、小人壓力值。 |

## 優化與溝通技巧：增強 AI 代理人的決策質量

為了讓 AI 更有效地參與開發，開發者需要使用特定的溝通技巧與數據表示法。

## 提示詞工程（Prompt Engineering）與少樣本學習

在與 AI 代理人溝通時，應提供大量的「少樣本學習」（Few-shot learning）範例。例如，向 AI 展示如何通過自動化線路（Automation Wires）連接智慧電池與煤炭發電機，以防止能源浪費。

AI 應該被賦予特定的「角色」（Persona），例如「資深《缺氧》基地設計師」，這能迫使模型採用更專業的語氣與知識領域。在處理複雜邏輯（如處理電解水系統的熱量）時，要求 AI 進行「鏈式思考」（Chain-of-Thought），逐步解釋其設計邏輯。

## 建立異常處理與自我修正機制

自動化過程中最常遇到的問題是代碼崩潰或邏輯死鎖。例如，小人可能會把自己關在牆後 suffocate。代理人必須具備「自我反思」（Reflection）的能力。

當觀測到「Duplicant Stress Rising」或「Oxygen Level Critical」時，代理人應優先中斷當前項目，切換到危機處理模式。開發者可以透過在溝通協議中加入「事件觸發器」（Event Triggers），當特定數值超過閾值時，強制 AI 進行環境掃描並更新優先級。

## 技術挑戰與風險控管

在開發此類高度自動化系統時，必須預見並處理潛在的技術風險。

## 內存洩漏與效能下降

《缺氧》本身存在嚴重的內存管理問題，長時間運行後內存佔用可能超過 12-13 GB。自動化腳本若頻繁調用反射（Reflection）或大規模序列化 Grid 數據，會加速崩潰過程。建議使用高性能的採樣庫（如 LiteNetLib 進行二進制傳輸）並定期重新啟動遊戲實例。

## 遊戲版本更新的相容性

由於《缺氧》持續進行質量改進更新（QOL Updates），每次更新都可能改變私有成員的名稱或方法簽名。為了讓 AI 能協助維護項目，開發者應使用自動化腳本提取最新的 Assembly-CSharp.dll 並生成最新的 API 文檔供 AI 參考。

## 總結與行動建議

構建《缺氧》自動化代碼是一個融合了逆向工程、分布式通訊與 Agentic AI 的前沿項目。開發者應優先建立穩定的 C# 數據導出模組，確保數據流的結構化與實時性。隨後，透過 MCP 協議將決策邏輯外包給具備 ReAct 模式的 AI 代理人團隊。

在與 AI 溝通時，核心在於精確建模遊戲的物理法則（如導熱係數與比熱容）與調度邏輯（如優先級系統）。透過不斷迭代「觀察-思考-行動」的閉環，自動化程序將能夠從簡單的資源收集演進到複雜的基地工業化管理。這不僅能大幅提升開發效率，更能探索出 AI 在複雜系統模擬中的極限。
